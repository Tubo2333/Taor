## 致命（Fatal）— 运行时崩溃或数据丢失

### F1. fireOnError 传递 broken context，onError handler 必然崩溃

**位置**：[hooks/src/registry.ts:319](packages/hooks/src/registry.ts#L319)

```typescript
await entry.handler({ session: null, turn: null, shared: null } as any, errors[0]!)
```

`fireOnError` 是 HookRegistry.execute() 收集到 handler 错误后触发的恢复路径。它构造的 SessionContext 全部字段为 `null`。任何 onError handler 如果尝试 `ctx.session.id` 或 `ctx.shared.loadedResources.get(...)` 都会抛出 `TypeError: Cannot read properties of null`。

由于 `fireOnError` 本身 catch 了异常（line 323: `} catch { /* Swallow */ }`），这个崩溃是**静默的**——handler 安装成功但永远收不到有效上下文。TG0 的 onError hook 不执行 ErrorRecovery，所以实际影响有限；但一旦 TG1 启用 ErrorRecovery，所有由 hook 内部错误触发的恢复都将静默失败。

**修复方案**：fireOnError 需要接收真实的 SessionContext。方案 A：在 execute() 参数中前置传入 ctx（如果 hook 签名包含 ctx）。方案 B：在 fireOnError 调用处构造最小有效 SessionContext `{ session: { id: "hook-error", ... }, shared: { loadedResources: new Map(), ... }, ... }`。

## 重要（Important）— 行为 bug、功能缺失、不一致

### I1. 结构 IAdapter.getModelInfo 返回类型与 canonical LLMAdapter.getModelInfo 不匹配

**位置**：[core/src/harness.ts:67](packages/core/src/harness.ts#L67) vs [adapters/src/types.ts:75](packages/adapters/src/types.ts#L75)

| 字段 | IAdapter 返回 | LLMAdapter 返回 (ModelInfo) |
|------|--------------|---------------------------|
| id | ✅ | ✅ |
| maxInputTokens | ✅ | ✅ |
| maxOutputTokens | ✅ | ✅ |
| provider | ❌ | ✅ |
| supportsThinking | ❌ | ✅ |
| supportsVision | ❌ | ✅ |
| supportsPromptCaching | ❌ | ✅ |
| supportsToolUse | ❌ | ✅ |
| costPer1kInput | ❌ | ✅ |
| costPer1kOutput | ❌ | ✅ |

TG0 中 harness.ts 从未调用 `getModelInfo()`，所以当前无运行时影响。但如果 TG1 的 CompressorPipeline 或 SubagentWorker 通过 IAdapter 接口调用 `getModelInfo` 并期望完整 ModelInfo，将得到 `undefined` 字段。

**修复方案**：将 IAdapter.getModelInfo 返回类型改为 `ModelInfo`（从 @taor/adapters import type），或保留当前子集但明确标注"TG0 subset, TG1 expand to full ModelInfo"。

### I2. IAdapter 缺少 `version`、`supports()`、`parseToolCalls()`、`countRequestTokens()` 方法

**位置**：[core/src/harness.ts:65-74](packages/core/src/harness.ts#L65-L74) vs [adapters/src/types.ts:71-101](packages/adapters/src/types.ts#L71-L101)

Canonical LLMAdapter 有 11 个成员，IAdapter 只声明了 7 个。缺失的 4 个中 `supports()` 是 SubagentWorker 和 CompressorPipeline 选择策略时的关键查询接口。

**修复方案**：TG1 中 SubagentWorker/CompressorPipeline 使用 IAdapter 时将需要 `supports()`。建议在 IAdapter 中补全缺失方法，或文档说明 IAdapter 是"TAOR loop 最小接口"，其他子系统应使用完整的 LLMAdapter 类型。

### I3. IPermissionEngine.removeRule 返回值类型不匹配

**位置**：[core/src/harness.ts:128](packages/core/src/harness.ts#L128) vs [permission/src/engine.ts:338](packages/permission/src/engine.ts#L338)

- 结构接口：`removeRule(pattern: string): void`
- Canonical：`removeRule(pattern: string): number`（返回被删除的规则数量）

`as any` 桥接导致返回值被静默丢弃。调用方无法得知是否有规则被删除。

**修复方案**：将 `IPermissionEngine.removeRule` 返回类型改为 `number`。

### I4. ISubagentCoordinator.spawn spec 缺少 `schema?: z.ZodType` 字段

**位置**：[core/src/harness.ts:176-182](packages/core/src/harness.ts#L176-L182) vs [subagent/src/types.ts:9-25](packages/subagent/src/types.ts#L9-L25)

Canonical SubagentSpec 有 `schema?: z.ZodType` 字段用于 StructuredOutput。通过 ISubagentCoordinator 接口 spawn 时此字段被丢弃——子 agent 将收不到 schema 约束。

**修复方案**：在 ISubagentCoordinator.spawn spec 中增加 `schema?: unknown`（避免 import zod），SubagentCoordinator.spawn 内部做类型收窄。

### I5. ISubagentCoordinator spawn done() 返回缺少 `artifacts?: Artifact[]` 字段

**位置**：[core/src/harness.ts:189-195](packages/core/src/harness.ts#L189-L195) vs [subagent/src/types.ts:43-49](packages/subagent/src/types.ts#L43-L49)

Canonical SubagentResult 有 `artifacts?: Artifact[]` 字段。通过结构接口获取的 done() 结果缺少此字段——子 agent 产物丢失。

**修复方案**：在 ISubagentCoordinator 的 done() 返回类型中增加 `artifacts?: Artifact[]`。

### I6. IMemoryFacade list() opts 缺少 `limit`/`offset` 分页参数

**位置**：[core/src/harness.ts:209](packages/core/src/harness.ts#L209) vs [memory/src/types.ts:33](packages/memory/src/types.ts#L33)

- 结构接口：`list(opts?: { prefix?: string; tags?: string[] }): Promise<{ key: string; value: unknown }[]>`
- Canonical MemoryStore.list：`list(opts?: { prefix?: string; tags?: string[]; limit?: number; offset?: number }): Promise<MemoryEntry[]>`

返回值类型也不同：结构接口返回简化的 `{ key, value }[]`，canonical 返回 `MemoryEntry[]`（含 metadata）。

**修复方案**：统一 IMemoryFacade list 签名与 MemoryStore 对齐，或明确文档说明 IMemoryFacade 是简化视图。

### I7. SubagentCoordinator 不调用 onError hook 处理 Worker 错误

**位置**：[subagent/src/coordinator.ts:193-198](packages/subagent/src/coordinator.ts#L193-L198)

```typescript
.catch((err) => {
  handle._onError({
    code: "execution_error",
    message: err instanceof Error ? err.message : String(err),
    subagentId: id,
  })
})
```

Worker 异常被捕获后只调用 `handle._onError()`（内部状态机 transition），不调用任何 hook。对比 adapter 错误在 harness.ts 中会触发 `onError` hook（line 660-672），子 agent 的错误处理不一致。

**修复方案**：在 `.catch()` 中添加 `this.hookRegistry?.execute("onError", ctx, error)`。

### I8. 工具执行错误不触发 onError hook

**位置**：[core/src/harness.ts:886-907](packages/core/src/harness.ts#L886-L907)

ACT phase 中 `tool.execute()` 抛出的异常被 catch，构建错误 ToolCallResult 后继续。与 adapter 错误处理（line 654-682，触发 onError hook）不一致。onError hook 在 adapter 错误时触发，在工具错误时不触发。

**修复方案**：在工具执行 catch 块中添加 `this.hookRegistry?.execute("onError", ...)` 调用，或明确文档说明 onError 只覆盖 adapter/harness 层错误，工具错误通过 tool-result 事件传播。

### I9. beforeSpawn/afterSpawnResult hooks 是 fire-and-forget

**位置**：[subagent/src/coordinator.ts:128-131](packages/subagent/src/coordinator.ts#L128-L131) 和 [subagent/src/coordinator.ts:189-191](packages/subagent/src/coordinator.ts#L189-L191)

```typescript
// beforeSpawn: not awaited
this.hookRegistry.execute("beforeSpawn", spec)

// afterSpawnResult: not awaited
this.hookRegistry.execute("afterSpawnResult", handle, result)
```

两个 hook 调用都没有 `await`。如果 handler 是 async 函数，其 Promise rejection 将被静默吞没（unhandled promise rejection）。`beforeSpawn` hook 的返回值（可能修改 spec）也被忽略。

**修复方案**：TG1 使 spawn() 变为 async，await 两个 hook 调用。TG0 建议至少记录 TODO 说明已知风险。

### I10. CompressorPipeline.compress() 未自动触发——上下文无限增长

**位置**：[core/src/harness.ts](packages/core/src/harness.ts)（TAOR 循环中无 compress 调用点）

`ICompressorPipeline` 通过 `setCompressor()` 注入，`CompressorPipeline.compress()` 已完整实现（trim + truncate）。但 TAOR 循环中**没有任何地方调用它**。消息数组 `this.messages` 在每个 turn 只增不减，长会话上下文将无限增长直到超过 LLM 上下文窗口。

**修复方案**：在 TAOR 循环的 OBSERVE 阶段后（turn-ended 事件前）添加 token 预算检查，超过 `triggerThreshold` 时调用 `this.compressorPipeline?.compress(ctx)`。由于 TG0 已有 trim + truncate 实现，此修复可以在 TG0 内完成。

### I11. JsonStore dirty 数据在进程退出时丢失

**位置**：[memory/src/store.ts:206-210](packages/memory/src/store.ts#L206-L210)

`flush()` 方法是唯一的持久化路径，但没有任何地方自动调用它：
- 没有 `process.on("exit", ...)` / `process.on("SIGINT", ...)` 处理器
- 没有定时器驱动的自动 flush
- 唯一调用 `flush()` 的是 MemoryFacade（但目前未实现）

如果进程非正常退出或用户忘记调用 `flush()`，所有 dirty 数据丢失。

**修复方案**：在 JsonStore 构造函数中注册 `process.on("exit", () => this.flush())` 和 `process.on("SIGINT", ...)`。注意 `process.on("exit")` 只能执行同步操作——需在注册时预先 flush 或使用 `process.on("beforeExit", ...)`。

## 建议（Suggestions）— 改进建议

### S1. IToolRegistry 缺少 `has(name: string): boolean` 方法

**位置**：[core/src/harness.ts:92-99](packages/core/src/harness.ts#L92-L99) vs [tools/src/registry.ts:147](packages/tools/src/registry.ts#L147)

Canonical ToolRegistry 有 `has()` 方法但 IToolRegistry 没有声明。TG0 中 harness.ts 不使用 `has()`（直接用 `get()` + undefined check），但如果未来子系统通过接口查询工具存在性，将报错。

### S2. 结构 ToolDef 缺少 canonical ToolDescriptor 的若干字段

**位置**：[core/src/harness.ts:47-55](packages/core/src/harness.ts#L47-L55) vs [tools/src/types.ts:129-145](packages/tools/src/types.ts#L129-L145)

ToolDef 缺少：`timeout`, `retry`, `onBeforeExecute`, `onAfterExecute`, `onError`, `permissions` 的类型不同（`string[]` vs `PermissionHint[]`）。TG0 harness 不需要这些字段，但 TG1 可能需要。

### S3. ISubagentCoordinator 返回类型缺少 `on()` 事件订阅方法

**位置**：[core/src/harness.ts:183-196](packages/core/src/harness.ts#L183-L196) vs [subagent/src/types.ts:27-41](packages/subagent/src/types.ts#L27-L41)

Canonical SubagentHandle 有 5 个 `on()` 重载（started, done, error, heartbeat, status-change）。结构接口完全没有。如果外部代码通过 `harness.spawn().on("done", ...)` 监听，将获得类型错误。

### S4. IMemoryFacade 三层完全重复——可提取 ILayerStore

**位置**：[core/src/harness.ts:203-228](packages/core/src/harness.ts#L203-L228)

user/project/session 三层接口完全相同（6 个方法 × 3 层 = 18 个方法声明）。可以提取为 `IMemoryLayer` 接口 + 三个 `readonly` 属性。

### S5. 文档声称 8 处 `as any` 但实际为 14 处

**位置**：[engine/src/index.ts:74](packages/engine/src/index.ts#L74)

代码注释说"8 次 `as any` 桥接"，但实际审查发现 14 处：
- 8 处 subsystem 注入（adapter, registry, permission, hooks, subagent, subagent adapter, subagent registry, subagent hookRegistry, memory, compressor = 10 处 `harness.*/subagent.* as any`）
- 4 处 stub→real config 转换（`as unknown as Partial<Config>`）

**修复方案**：更新注释中的数字或使用脚本自动计数。

### S6. Worker 中 formatToolResult 返回值被强制转换为 string

**位置**：[subagent/src/worker.ts:228-233](packages/subagent/src/worker.ts#L228-L233)

```typescript
content: formattedContent as string,
```

`adapter.formatToolResult()` 可能返回结构化对象（非字符串），强制转换为 string 后可能产生 `[object Object]`。不过当前 AnthropicAdapter 返回的是 string，所以暂时安全。

### S7. CompressorPipeline cache 无 LRU/驱逐策略——无限增长

**位置**：[compressor/src/pipeline.ts:51](packages/compressor/src/pipeline.ts#L51) 和 [compressor/src/pipeline.ts:117](packages/compressor/src/pipeline.ts#L117)

`this.cache = new Map<string, CompressedContext>()` 只有 `set()` 和 `clear()`，没有容量限制或 LRU 驱逐。长会话中 cache 无限增长。

**修复方案**：添加 maxCacheSize（如 64）和 LRU 驱逐，或使用 WeakMap/TTL 过期。

### S8. TAOR 循环缺少心跳机制

**位置**：[core/src/harness.ts](packages/core/src/harness.ts)（无 heartbeat 相关代码）

HeartbeatEvent 类型已定义（events.ts:116-122），SubagentWorker 有 turn-boundary heartbeat（worker.ts:243-247），但主 TAOR 循环没有心跳发射器。长时间运行的工具执行期间，消费者无法获知 liveness。

**修复方案**：TG1 中为长工具执行添加定时心跳（setInterval → pushEvent(heartbeat) → clearInterval）。

### S9. SubagentCoordinator 内部结构接口的 stopReason 类型为 `string`

**位置**：[subagent/src/coordinator.ts:29](packages/subagent/src/coordinator.ts#L29)

CoordinatorAdapter.think() 的 stop event 中 `reason?: string`，而 canonical ThinkEvent 的 stop.reason 是 `StopReason`（联合类型：`"end_turn"|"max_tokens"|...`）。不影响运行时。

### S10. SubagentCoordinator 不引用 @taor/hooks 的 tsconfig reference

**位置**：[subagent/tsconfig.json](packages/subagent/tsconfig.json)

Coordinator 调用 `hookRegistry.execute()` 但通过结构接口（`{ execute(hook, ...args): Promise<unknown[]> }`），避免了对 `@taor/hooks` 的编译期依赖。这是**正确的设计**——不添加 reference 是正确的。

### S11. CompressorPipeline cacheKey 参数使用 `any[]`

**位置**：[compressor/src/pipeline.ts:86,135](compressor/src/pipeline.ts#L86-L135)

两处 `cacheKey(messages as any, ...)` 可以改用 `Message[]` 类型，因为 `messages` 来自 `ctx.turn.messages` 已经有完整类型。

### 维度 1：结构接口一致性（7.5/10）

**审查方法**：逐一对比 harness.ts 中 6 个结构接口（IAdapter/IToolRegistry/IPermissionEngine/IHookRegistry/ISubagentCoordinator/IMemoryFacade/ICompressorPipeline）与其对应的 canonical 类型的所有字段。

**结论**：核心方法签名在运行时均兼容（`as any` 桥接 + 结构子集原则）。存在 5 处字段级遗漏和 1 处返回类型不匹配，均为非致命但在 TG1 扩展时可能成为 bug。

## 结论

TG0 全 12 步交付了一个架构良好、结构清晰、零编译错误的 agent 框架基础。依赖反转设计（6 个结构接口 + 14 处 `as any` 桥接）有效解决了 9 包 monorepo 中的循环引用问题。TAOR 主循环的 7 个 hook 点集成正确，资源管理（AbortController/Promise resolver/event listener）遵循了正确的 capture-null-resolve 模式。

主要技术债务集中在三个方面：(1) 错误传播不一致——5 种错误来源只有 2 种触发 onError hook；(2) compress() 已实现但未集成进 TAOR 循环；(3) fireOnError 的 broken context 在 TG1 启用 ErrorRecovery 后会静默失败。

建议优先完成 TG0.1 的 5 项 P0/P1 修复（~40 行代码），然后将接口对齐（I1-I6）纳入 TG1 规划。
